The Problem

Why do we need standards like SOS?

Without Standards

  • Every sensor vendor uses a different API
  • Data formats incompatible across systems
  • No way to query sensors from multiple sources uniformly
  • "Vendor lock-in"

With OGC SOS 2.0

  • Universal operation names: GetCapabilities, DescribeSensor, GetObservation
  • Standardised XML responses (O&M)
  • Any compliant client can talk to any compliant server
  • Interoperability by design
GeoPulse implements the SOS 2.0 operation structure on top of real historical weather data, making it queryable by any OGC-aware tool.

The OGC Standards Stack

Three standards working together

SOS

Sensor Observation Service


Defines how to ask for sensor data — the URL operations, request/response structure


Three core operations:
GetCapabilities
DescribeSensor
GetObservation

O&M

Observations & Measurements


Defines what a reading looks like:
• who observed (sensor)
• what was measured
• where (location)
• when (timestamp)
• result (value + unit)

SensorML

Sensor Model Language


Defines the sensor's identity:
• Unique identifier
• Physical location (GML Point)
• Output capabilities (what it measures)
• Units of measurement

GetObservation — The Filter Engine

GET /sos/observations — all filters are combinable

🌍 Spatial

  • country=Japan
  • bbox=60,10,140,50
    minLon,minLat,maxLon,maxLat

⏱ Temporal

  • after=2025-01-01
  • start=2025-01-01
  • end=2025-03-31

📊 Parameter

  • param=temperature
  • op=gt
  • value=30
  • Operators: eq neq lt lte gt gte between
Example: all stations in Asia where temperature was between 30–40°C in Jan 2025
/sos/observations?bbox=60,5,150,55&param=temperature&op=between&value=30&value2=40&start=2025-01-01&end=2025-01-31

Dynamic SQL — Safe Parameterized Queries

How filters become database queries

conditions = []
bind_params = []

if country:                                    # spatial
    conditions.append("LOWER(country) = LOWER(?)")
    bind_params.append(country)

if start:                                      # temporal
    conditions.append("timestamp >= ?")
    bind_params.append(start)

if param in ALLOWED_PARAMS and op == "between":  # parameter
    conditions.append(f"{param} BETWEEN ? AND ?")
    bind_params += [value, value2]

sql = f"SELECT * FROM observations WHERE {' AND '.join(conditions)} LIMIT ?"
rows = conn.execute(sql, bind_params).fetchall()
Security: ? placeholders prevent SQL injection. User input never gets pasted into the SQL string directly. param is validated against an allowlist before use as a column name.

O&M — Observations & Measurements

How our database maps to the O&M standard

O&M Concept Definition Our Database Column
featureOfInterest The place being observed location_name, latitude, longitude
observedProperty What is being measured temperature / humidity / wind_speed / ...
procedure The sensor that made the observation sensor_id → entry in SensorML
phenomenonTime When the observation was made timestamp
result The measured value + unit numeric column value + unit from SensorML
The XML response from /sos/observations wraps each row in <Observation> tags, directly reflecting the O&M model.

Backend — FastAPI

Python web framework — each URL is a decorated function

@app.get("/sos/capabilities")
def get_capabilities():
    # returns static Capabilities XML
    return Response(content=xml,
        media_type="application/xml")

@app.get("/sos/sensor")
def describe_sensor():
    # reads sensorml.xml from disk
    with open("sensorml.xml") as f:
        return Response(content=f.read(),
            media_type="application/xml")

@app.get("/sos/sensors")
def get_sensors():
    # returns 25 sensor positions as JSON
    rows = conn.execute("""
        SELECT sensor_id, location_name,
        AVG(lat), AVG(lon) FROM observations
        GROUP BY sensor_id
    """).fetchall()
    return [dict(r) for r in rows]

Full Request Lifecycle

You set filters and click "Get Observations" — here's every step

1. Browser reads form → builds URL: /sos/observations?country=Japan&param=temperature&op=gt&value=25
↓ HTTP GET
2. FastAPI receives params → builds SQL → SELECT * FROM observations WHERE LOWER(country)=LOWER('Japan') AND temperature > 25 LIMIT 500
↓ SQLite executes
3. rows_to_xml() → wraps each row in <Observation> tags → returns application/xml
↓ back to browser
4. DOMParser parses XML → JS array of objects → simultaneously feeds map markers, table rows, and Chart.js
